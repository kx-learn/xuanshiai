# 会员体系和积分体系设计

> 版本：v1.0
>
> 适用范围：首页推荐、我的页面、关系互动、社区、纸飞机和后续商业化模块。
>
> 本文依据以下文档整理：

> - `docs/后端功能模块任务划分清单.md`

## 1. 设计结论

当前采用“一种付费会员等级、三种购买周期”的设计：

| 用户类型 | 说明 |
| --- | --- |
| 普通用户 | 不购买会员也可以使用基础功能，但受每日额度和可见范围限制 |
| VIP 会员 | 只有一个付费等级，按月、季度、年度购买；周期不同不改变权益，只改变有效期和价格 |

暂不设计 SVIP、黄金会员、钻石会员等多个同时生效的等级。以后如果需要分级，可以通过 `membership_level` 和权益配置扩展，不修改业务接口。

积分与会员是两套独立体系：

- 会员来自购买、运营赠送或邀请奖励，核心是“时间权益和额度权益”。
- 积分来自签到、任务、邀请和运营奖励，核心是“可消费的虚拟账户余额”。
- 会员不能直接替代积分，积分也不能默认替代现金或会员，除非具体商品明确支持。

## 2. 会员解决什么问题

会员的作用不是绕过平台规则，而是提高正式用户的使用效率和曝光能力：

1. 提高每日申请、爆灯、浏览等有限资源的额度。
2. 允许用户查看普通用户不能查看的访客和历史记录。
3. 允许用户使用高级筛选和更完整的公开资料字段。
4. 提供会员标识和合理的推荐曝光权益。
5. 为后续积分兑换、置顶曝光和增值服务提供统一的权益判断入口。

会员始终不能绕过以下规则：

- Access Token、账号状态和安全审计。
- 手机号绑定要求。
- 资料完整度要求。
- 实名认证要求。
- 双方拉黑、目标隐私设置和内容安全规则。
- 申请、聊天、举报和支付的资源归属校验。

## 3. 会员套餐

### 3.1 套餐类型

现有 `config_membership_package` 已支持套餐配置，推荐使用以下三个稳定编码：

| `code` | 名称 | 建议有效期 | 说明 |
| --- | --- | ---: | --- |
| `monthly` | 月度 VIP | 30 天 | 适合首次购买和短期体验 |
| `quarterly` | 季度 VIP | 90 天 | 默认推荐套餐，日均价格低于月度 |
| `yearly` | 年度 VIP | 365 天 | 长期用户套餐，日均价格最低 |

有效期使用服务端天数计算，不使用前端传入的结束时间。

### 3.2 会员权益

以下数值与当前配置和首页任务保持一致。所有数值必须配置化，不能写死在前端。

| 权益编码 | 普通用户 | VIP 会员 | 后端使用位置 |
| --- | ---: | ---: | --- |
| `apply_daily_limit` | 3 次/日 | 10 次/日 | 申请认识 |
| `superlike_daily_limit` | 1 次/日 | 3 次/日 | 爆灯 |
| `browse_daily_limit` | 20 次/日 | 不限制 | 他人完整资料浏览 |
| `browse_high_match_bonus` | 额外 5 次 | 不适用 | 高匹配分浏览奖励 |
| `browse_history_scope` | 仅当天 | 不限历史 | 我的浏览记录 |
| `visitor_detail` | 只能看数量 | 可以看访客名片 | 我的访客 |
| `detail_visibility` | 受普通用户字段限制 | 可查看 VIP 字段 | 他人主页 |
| `advanced_filter` | 不开放 | 开放 | 首页筛选 |
| `vip_badge` | 无 | 有 | 名片和主页标识 |
| `pure_free_filter` | 可读取基础结果 | 可使用完整筛选能力 | 推荐/广场 |
| `priority_exposure` | 无 | 按配置决定 | 推荐排序，不得保证固定曝光 |

“不限制”只表示不消耗该项普通额度，不表示可以绕过 Redis 故障、账号封禁、隐私设置或频率保护。

### 3.3 会员资格判断

服务端统一使用以下条件判断有效会员：

```sql
status = 1
AND (start_at IS NULL OR start_at <= UTC_TIMESTAMP())
AND (end_at IS NULL OR end_at > UTC_TIMESTAMP())
```

禁止以前端提交的 `is_vip`、`package_type`、`expires_at` 或剩余次数作为权限依据。

推荐封装统一服务：

```text
MembershipService.is_active(user_id)
MembershipService.get_rights(user_id)
MembershipService.get_effective_limit(user_id, quota_code)
MembershipService.require_right(user_id, right_code)
```

推荐结果、主页详情、访客、申请和爆灯都必须调用同一套服务，避免不同接口判断不一致。

## 4. 会员状态和购买规则

### 4.1 会员状态

`user_membership.status` 建议固定为：

| 状态 | 含义 | 可否使用权益 |
| ---: | --- | --- |
| `0` | 待生效 | 否 |
| `1` | 生效中 | 是 |
| `2` | 已过期 | 否 |
| `3` | 已退款/撤销 | 否 |

现有表默认状态为 `1`，支付订单接入后应改为：支付成功才写入生效记录。运营赠送可以直接写入 `1`，但必须有操作来源和审计记录。

### 4.2 续费、叠加和退款

建议规则：

1. 首次购买：`start_at = max(now, 当前有效会员 end_at)`，`end_at = start_at + duration_days`。
2. 未过期续费：顺延有效期，不覆盖原有效期。
3. 已过期续费：从当前时间重新开始。
4. 同一订单只能生成一次会员权益。
5. 支付回调重复到达时，不得重复延长会员时间。
6. 退款成功后，未使用权益按订单规则撤销；具体是否按剩余天数退款需要财务确认。
7. 同一用户可以存在历史多条会员记录，但同一时间只有一个有效权益计算结果。

### 4.3 会员购买数据流

```text
查询在售套餐
  -> 创建待支付订单
  -> 第三方支付
  -> 服务端验签回调
  -> 锁定订单
  -> 校验订单金额、用户、商品和状态
  -> 更新订单为支付成功
  -> 创建或延长 user_membership
  -> 写入通知和操作流水
```

支付回调必须以第三方签名和服务端订单金额为准，不能信任前端金额或会员天数。

## 5. 积分解决什么问题

积分用于低金额、低风险的虚拟权益兑换和用户活跃激励，主要目标是：

- 鼓励用户完成资料、认证、签到和邀请。
- 让用户通过持续参与获得部分增值权益。
- 为纸飞机资料解锁、曝光卡、额外次数等二期能力提供统一消费账户。
- 形成可审计的收入与消耗流水。

积分不是现金，不支持用户之间转账，不直接提现，不允许前端修改余额。

## 6. 积分获得方式

下表是建议的初始配置，不应直接写死在 Service 中。正式上线前由运营确认数值和每日上限。

| 来源编码 | 获得场景 | 建议积分 | 限制 | 幂等依据 |
| --- | --- | ---: | --- | --- |
| `checkin` | 每日首次签到 | 5 | 每个用户每天 1 次 | `user_checkin(user_id, checkin_date)` |
| `profile_complete` | 资料达到 100% | 50 | 同一用户只奖励 1 次 | `user_task(user_id, task_code)` |
| `realname_verified` | 实名认证通过 | 100 | 同一用户只奖励 1 次 | 认证记录 ID |
| `upload_first_photo` | 首次上传审核通过相册图 | 20 | 同一用户只奖励 1 次 | 媒体审核记录 ID |
| `first_match` | 首次匹配成功 | 30 | 同一用户只奖励 1 次 | 匹配记录 ID |
| `first_chat` | 首次有效聊天 | 10 | 同一用户只奖励 1 次 | 会话/消息事件 ID |
| `invite_register` | 邀请用户完成注册 | 20 | 每日/每月上限 | `invite_record` 状态变更 |
| `invite_realname` | 被邀请用户完成实名 | 100 | 邀请关系有效且未奖励 | `invite_record` 状态变更 |
| `operation_reward` | 客服、活动或后台补发 | 配置 | 必须填写原因 | 后台操作 ID |

### 6.1 积分不应奖励的行为

以下行为不能直接奖励积分，避免刷量：

- 重复浏览同一用户。
- 重复收藏和取消收藏。
- 重复申请认识。
- 给自己点赞、评论或发送纸飞机。
- 仅创建支付订单但未支付。
- 仅提交认证材料但未审核通过。
- 使用同一设备批量注册邀请账号。

风控系统可以延迟发放积分，审核通过后再入账。

## 7. 积分可以做什么

建议按一期基础能力和二期商业化能力分开：

### 7.1 一期可预留，不立即开放

| 消费编码 | 用途 | 建议消耗 | 说明 |
| --- | --- | ---: | --- |
| `extra_apply` | 额外申请认识次数 | 20/次 | 不能绕过实名和账号状态 |
| `extra_superlike` | 额外爆灯次数 | 50/次 | 仍需实名、绑定手机号和目标可见 |
| `browse_unlock` | 额外浏览额度 | 10/次 | 不能绕过隐私设置 |
| `exposure_card` | 额外曝光卡 | 按商品配置 | 只能提高候选排序权重，不能保证曝光 |

### 7.2 二期明确能力

| 消费编码 | 用途 | 规则 |
| --- | --- | --- |
| `paper_plane_unlock` | 纸飞机资料解锁 | 同一对话、同一用户只扣一次；有效期由商品配置 |
| `profile_detail_unlock` | 某个资料字段解锁 | 必须记录目标用户、字段、消耗和有效期 |
| `membership_exchange` | 积分兑换会员天数 | 是否开放、兑换比例和上限待定 |
| `service_coupon` | 红娘或咨询服务优惠 | 需要订单、核销和退款规则 |

积分兑换不能直接修改会员表或额度 Redis。正确流程是创建权益订单/兑换记录，事务成功后再发放具体权益。

## 8. 积分账本设计

### 8.1 现有表复用

项目已有 `user_points`：

- `user_id`
- `type`
- `amount`
- `balance`
- `desc`
- `created_at`

该表可以作为积分流水基础，但要满足幂等、审计和并发要求，建议补充：

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `biz_type` | varchar(32) | `checkin/task/invite/redeem/refund/adjust` |
| `biz_id` | varchar(64) | 业务事件 ID |
| `idempotency_key` | varchar(128) | 防止重复发放或扣除 |
| `before_balance` | int | 变更前余额 |
| `after_balance` | int | 变更后余额，替代仅依赖 amount 反推 |
| `operator_id` | bigint/null | 后台调整时的操作人 |
| `expire_at` | datetime/null | 如果积分有有效期，记录到期时间 |
| `status` | tinyint | `1` 有效、`2` 冻结、`3` 已过期、`4` 已冲正 |

建议唯一约束：

```text
UNIQUE(user_id, idempotency_key)
```

### 8.2 入账事务

积分增加和扣除必须使用同一个事务：

```text
开始事务
  -> 锁定用户积分账户/查询当前余额 FOR UPDATE
  -> 检查 idempotency_key 是否已成功处理
  -> 检查余额是否足够
  -> 计算 before_balance / after_balance
  -> 写入 user_points
  -> 更新或创建积分账户余额
  -> 写入业务权益记录
提交事务
```

余额不足返回 `409` 或项目统一的业务错误码，不能返回负余额。

### 8.3 积分有效期

推荐暂定“积分永久有效”，先不实现积分过期，原因是当前已有表不是分批次余额模型，直接过期容易造成账目不一致。

如果后续需要过期，必须改为批次余额或 FIFO 消耗：

1. 每次入账形成独立积分批次。
2. 消费优先扣除最早到期批次。
3. 定时任务将过期余额转为过期流水。
4. 用户查询余额时不能只做 `SUM(amount)`，必须考虑批次状态。

## 9. 推荐数据库模型

### 9.1 会员相关

复用：

- `config_membership_package`：在售套餐和权益配置。
- `user_membership`：用户会员有效期和历史记录。
- `payment_order`：支付订单和支付状态。

建议补充或确认：

- `user_membership.package_id`：关联具体套餐，避免只靠 `package_type`。
- `user_membership.source`：`payment/gift/invite/operation/exchange`。
- `user_membership.grant_id`：赠送或兑换业务单号。
- `user_membership.rights_snapshot`：购买时权益快照，防止后台修改套餐后影响历史订单解释。
- `payment_order.idempotency_key`：防重复创建订单。

### 9.2 积分相关

复用：

- `user_points`：积分流水。
- `user_task`：任务完成和领奖状态。
- `user_checkin`：签到记录。
- `invite_record`：邀请关系和奖励状态。

建议新增：

- `user_point_account`：用户当前可用余额，按 `user_id` 唯一。
- `point_product`：积分兑换商品和消耗数量。
- `point_redeem_order`：积分兑换订单、状态、幂等键和核销信息。
- `user_point_hold`：需要审核或延迟发放时冻结的积分奖励。

## 11. 错误和状态约定

| HTTP 状态 | 场景 | 前端处理 |
| ---: | --- | --- |
| `401` | 未登录或 Token 无效 | 跳转登录 |
| `403` | 未绑定手机号、未实名、账号停用、无权限 | 跳转绑定/认证/申诉 |
| `404` | 套餐、订单或兑换商品不存在 | 刷新页面 |
| `409` | 余额不足、订单状态冲突、重复领取 | 刷新当前状态，不重复提交 |
| `422` | 套餐编码、任务编码或参数格式错误 | 展示字段错误 |
| `429` | 签到、任务或邀请达到频率限制 | 显示下次可操作时间 |
| `503` | 支付、Redis 或第三方服务不可用 | 稍后重试，不展示内部异常 |

错误响应不能包含数据库 SQL、支付密钥、Token、完整身份证号或内部堆栈。

## 12. 权益检查顺序

所有涉及会员、积分、额度的接口统一按以下顺序执行：

1. 校验 Token 和账号状态。
2. 校验手机号绑定和实名认证等基础门槛。
3. 校验资源归属、隐私和拉黑关系。
4. 查询服务端有效会员和权益快照。
5. 查询或扣减 Redis 日额度。
6. 锁定数据库资源并执行积分/订单/关系写入。
7. 写入通知和审计流水。
8. 事务提交成功后返回结果。

不能先扣积分再验证目标用户，也不能先发通知再提交会员或积分流水。

## 13. 验收清单

### 13.1 会员

- [ ] 普通用户和 VIP 的每日申请次数分别由服务端配置决定。
- [ ] 普通用户和 VIP 的爆灯次数分别由服务端配置决定。
- [ ] 普通用户只能查看当天浏览记录，VIP 可查看历史记录。
- [ ] 普通用户看不到访客详情，VIP 可以查看访客名片。
- [ ] 会员不能绕过手机号、实名认证、隐私、拉黑和封禁规则。
- [ ] 同一订单重复支付回调不会重复发放会员。
- [ ] 续费不会覆盖未过期会员时间。
- [ ] 退款、过期和撤销后的权益立即失效。
- [ ] 订单只能由所有者查询，支付回调必须验签。

### 13.2 积分

- [ ] 重复签到不会重复发积分。
- [ ] 同一任务只能领取一次奖励。
- [ ] 邀请注册和邀请实名奖励按状态流转发放。
- [ ] 积分增加和扣除都有完整流水。
- [ ] 并发扣分不能产生负余额或重复扣分。
- [ ] 重试同一个业务事件不会重复入账。
- [ ] 积分兑换成功后才扣除积分。
- [ ] 兑换失败会回滚或冲正积分。
- [ ] 用户只能查看自己的余额、流水、任务和邀请记录。
- [ ] 后台调整积分必须记录操作人、原因和前后余额。

### 13.3 安全

- [ ] 前端不能提交会员等级、会员结束时间、积分余额或订单金额覆盖服务端数据。
- [ ] 所有奖励和消费接口都有幂等键或唯一业务事件约束。
- [ ] 支付密钥和第三方 App Secret 只从环境变量或配置中心读取。
- [ ] 生产响应和日志不输出支付参数、Token、密钥和敏感身份证明。
- [ ] 会员权益、积分余额和额度在数据库、Redis、订单状态之间保持一致。

## 14. 需要产品最终确认的事项

以下内容当前不能由后端自行决定：

1. 月度、季度、年度会员的价格和折扣。
2. 会员是否增加“优先曝光”、置顶、专属客服等权益。
3. 会员续费是否允许叠加，退款按整单还是剩余天数计算。
4. 会员到期后已经购买的曝光、解锁权益是否继续有效。
5. 各类签到、任务、邀请的具体积分数值和每日上限。
6. 积分是否永久有效，还是按批次过期。
7. 积分是否可以兑换会员、爆灯、额外申请、纸飞机资料解锁或曝光卡。
8. 申请被拒绝或过期时，是否退还额度或积分。
9. 同一用户重复邀请、设备关联邀请和异常邀请的奖励规则。
10. 纸飞机资料解锁的字段范围、积分消耗、有效期和是否需要对方同意。
11. 积分兑换是否需要支付差额、库存和人工审核。
12. 会员和积分首期是否只做后台配置/测试权益，暂不接入真实支付。

## 15. 推荐开发顺序

### 阶段一：配置和只读查询

1. 确认会员权益编码和积分来源/消费编码。
2. 补齐套餐权益配置和积分配置表。
3. 实现会员状态查询、积分余额查询和流水分页。
4. 接入 overview 的会员和积分摘要。

### 阶段二：非支付积分能力

1. 实现签到。
2. 实现资料、认证、首次匹配等任务奖励。
3. 实现邀请状态流转和奖励。
4. 增加积分账本锁、幂等和冲正。

### 阶段三：会员订单

1. 实现套餐下单和订单查询。
2. 接入支付沙箱。
3. 实现签名回调和会员延长。
4. 实现退款、过期和对账。

### 阶段四：积分消费和权益核销

1. 实现积分商品和兑换订单。
2. 实现额外申请、爆灯、曝光卡或纸飞机解锁。
3. 增加权益有效期、重复消费和退款冲正。
4. 增加运营后台、审计和财务报表。
