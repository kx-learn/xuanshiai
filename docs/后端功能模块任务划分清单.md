# 宣誓爱后端功能模块任务划分清单

## 1. 使用说明

本文档依据以下两份产品导图整理：

- `宣誓爱基本功能架构（清爽版）.xmind`：作为后端模块边界依据。
- `宣誓爱终极版.xmind`：作为完整需求来源。

只列后端任务。页面、交互、动画、客户端权限申请等不单独拆任务，但会转化为接口、状态、权限、异步任务和管理能力。

优先级定义：

- **P0**：一期上线阻塞项；涉及账号安全、合规、核心闭环或数据一致性。
- **P1**：一期重要能力；缺失会明显影响可用性，但不阻塞最小上线闭环。
- **P2**：二期增强或商业化能力。

分期定义：

- **一期**：登录、身份/资料、基础推荐、基础牵线、基础社区、匹配后聊天、通知、账号安全、基础后台和内容风控形成可运营闭环。
- **二期**：完整认证、AI/算法增强、付费服务、活动、积分任务、语音/实时能力和精细化运营。

## 2. 共享基础能力（先于七个业务模块）

### 2.1 账号、身份和权限基础

- **P0/一期**：用户、用户身份类型、账号状态、认证状态、资料状态、会员状态、注销状态数据模型。
- **P0/一期**：JWT access/refresh token、会话表、单设备/全设备退出、token 轮换和失效机制。
- **P0/一期**：用户端、红娘端、管理员端角色模型；管理员 RBAC、菜单/资源权限、数据范围权限。
- **P0/一期**：统一登录态、封禁/拉黑/注销/暂停服务等状态拦截中间件。
- **P0/一期**：统一错误码、分页、幂等键、审计字段、软删除和时区规则。

### 2.2 数据和异步基础

- **P0/一期**：MySQL 表结构、索引、唯一约束、外键/逻辑关联、迁移方案和回滚方案。
- **P0/一期**：Redis 用于验证码、限流、短期状态、未读计数和幂等控制；明确禁止把关键业务事实只存 Redis。
- **P0/一期**：文件上传抽象、类型/大小/数量校验、病毒或恶意文件扫描预留、访问授权和删除策略。
- **P1/一期**：事件总线或可靠异步任务，用于通知、内容审核、推荐刷新、媒体处理和第三方回调重试。
- **P0/一期**：操作审计日志、管理员操作日志、敏感字段脱敏日志、第三方调用日志和告警指标。

### 2.3 风控和内容安全公共服务

- **P0/一期**：统一风控决策接口：允许、二次验证、人工审核、拒绝、限流、封禁五类结果。
- **P0/一期**：设备/IP/账号/手机号/微信标识关联；登录、注册、发帖、评论、私信、申请认识和举报限频。
- **P0/一期**：违禁词词库、词条级别、命中策略（拦截、替换为 `*`、放行并记录、转人工）、词库版本和灰度发布。
- **P0/一期**：文本规范化后检测，覆盖昵称、简介、帖子、评论、聊天、纸飞机、红娘案例和反馈内容。
- **P1/一期**：图片/视频/语音审核适配层，支持同步初审、异步复审、人工复核和审核结果回调。
- **P0/一期**：用户举报、内容举报、举报证据、处理状态、处理结论、申诉、复审和证据留存。
- **P0/一期**：内容状态机：待审、可见、审核中、屏蔽、删除、申诉中、申诉通过/驳回；禁止用物理删除代替治理记录。

### 2.4 API 与验收基础

- **P0/一期**：所有接口按 `docs/api/` 规范编写完整请求/响应字段、权限、错误码、状态流转和变更记录。
- **P0/一期**：认证、权限、输入校验、并发、幂等、敏感信息脱敏的单元测试和接口测试。
- **P0/一期**：真实 MySQL/Redis 集成测试、跨用户授权测试、封禁/拉黑后的回归测试。
- **P1/一期**：慢查询、错误率、第三方失败率、审核积压、消息投递失败、关键业务漏斗监控。

## 3. 登录 + 首页模块

### 3.1 登录、注册和账号生命周期

- **P0/一期**：手机号验证码发送、校验、过期、重试、单手机号限频、注册/登录合并和新用户初始化。
- **P0/一期**：微信登录、openid/unionid 绑定、手机号与微信互绑、重复账号合并/冲突处理。
- **P0/一期**：注册身份选择：自己找对象、父母帮找、找搭子、推广红娘、合伙人、服务红娘；保存来源和版本。
- **P0/一期**：首次登录强制协议接受、未成年/年龄校验、资料完善引导状态。
- **P0/一期**：账号封禁、解封、注销申请、注销冷静期、数据匿名化/删除、关联会话失效。
- **P1/一期**：修改/换绑手机号、注销前二次验证、账号找回和异常登录处理。

### 3.2 身份认证

- **P0/一期**：实名认证申请、姓名/身份证号校验、成年校验、状态查询、失败原因、补交/重试。
- **P0/一期**：认证权限门禁：发帖、评论、点赞、关注、申请认识、聊天、纸飞机等能力按认证状态控制。
- **P1/一期**：人脸认证第三方服务适配层：请求签名、回调验签、超时/重试、幂等、结果落库、原始材料最小化留存。
- **P1/一期**：人脸服务供应商配置、沙箱/生产隔离、失败降级为人工审核，禁止生产环境使用 mock。
- **P2/二期**：学历、房产、单身承诺、婚姻状态等认证；人工审核、复审、认证有效期和认证历史。

### 3.3 首页推荐、搜索和浏览

- **P0/一期**：推荐候选池、资料可见性、年龄/城市/性别/婚姻/认证状态过滤、封禁和拉黑排除。
- **P0/一期**：单人推荐、广场卡片流、分页/游标、稳定排序、去重和每日刷新快照。
- **P0/一期**：基础筛选（城市、年龄、学历）；筛选条件校验、保存和清除。
- **P0/一期**：查看个人主页、浏览足迹、无痕浏览开关及会员权限校验。
- **P0/一期**：收藏、取消收藏、不感兴趣、分享数据记录、举报和拉黑入口。
- **P1/一期**：推荐理由、共同兴趣、认证标签、距离/IP 等展示字段的隐私过滤。
- **P1/一期**：谁看过我、我看过谁，会员查看期限和访客详情脱敏规则。
- **P2/二期**：AI 合拍度、MBTI/兴趣/行为加权、每日 12 点刷新、推荐实验、置顶曝光和会员专区。

## 4. 红娘（牵线）模块

### 4.1 红娘角色和资质

- **P0/一期**：红娘申请、资料提交、申请状态、审核意见、驳回重试和服务状态。
- **P0/一期**：专业红娘/热心红娘/推广红娘/合伙人角色区分、认证标识和权限边界。
- **P0/一期**：红娘主页：简介、擅长领域、服务价格、评分、成功案例、可见性和内容审核状态。
- **P1/一期**：红娘排行榜、成功牵线计数、评分统计、防刷和异常评价处理。
- **P1/一期**：红娘可服务城市、服务时间、接单上限、暂停接单和平台下架。

### 4.2 牵线服务

- **P0/一期**：服务红娘列表、详情、用户提交牵线申请、需求资料、匹配对象建议和申请状态。
- **P0/一期**：热心红娘介绍关系记录、被介绍人授权、申请次数/频控和重复介绍处理。
- **P1/一期**：1v1 情感咨询预约：日期、时段、咨询方式、订单、支付前后状态和取消规则。
- **P1/一期**：私人定制申请、套餐、专属红娘分配、需求分析、匹配、见面安排、跟进记录和完结评价。
- **P1/一期**：红娘与用户的服务会话、消息留痕、服务工单、转派、超时和投诉处理。
- **P2/二期**：支付、退款、佣金/结算、套餐权益、服务订单对账和财务流水。
- **P2/二期**：AI 红娘会话、模型调用、上下文隔离、敏感问题拦截、免责声明和人工转接。
- **P2/二期**：电话/微信联系方式交换申请、双方同意、脱敏展示、过期和撤回。

## 5. 社区模块

### 5.1 动态、话题和互动

- **P0/一期**：帖子发布、草稿/提交/审核/发布/屏蔽/删除状态；文字、图片基础媒体和话题关联。
- **P0/一期**：关注、同城、发现三类信息流；全部/热门/最新排序、分页、去重和屏蔽规则。
- **P0/一期**：帖子详情、评论、评论回复、点赞/取消点赞、作者删除和权限校验。
- **P0/一期**：帖子/评论举报、内容审核、敏感词替换为 `*`、管理员删除/屏蔽和用户可见提示。
- **P1/一期**：话题创建/编辑/上下线、热门排行、参与人数、浏览量和话题内容聚合。
- **P1/一期**：内容声明、双重认证门槛、用户隐私设置、同城/IP/公司/学校字段的展示控制。
- **P1/一期**：通知：评论、点赞、关注、系统治理结果及未读计数。

### 5.2 纸飞机和活动

- **P1/一期**：纸飞机匿名头像/昵称、文字发送、捡取、收回、回复、结束、举报、申请解锁和对话权限。
- **P1/一期**：纸飞机配额、积分消耗、重复捡取、对话时效、匿名身份与实名账号的隔离映射。
- **P2/二期**：语音纸飞机、语音消息、图片/拍摄、快捷话题和内容审核。
- **P2/二期**：线下活动发布、报名、审核、缴费、取消、签到、结束、地点延迟披露、客服和活动评价。
- **P2/二期**：活动报名费支付、退款、名额并发控制和活动内容后台运营。

## 6. 消息模块

### 6.1 认识申请与匹配

- **P0/一期**：申请认识创建、每日次数、认证门禁、会员/积分额度、幂等和重复申请限制。
- **P0/一期**：申请我的/我的申请列表、同意/拒绝、过期、撤回、拒绝后次数返还规则。
- **P0/一期**：同意后原子创建匹配关系和聊天会话；自动欢迎语只能发送一次。
- **P0/一期**：匹配解除、拉黑、举报、封禁对已有关系和会话的影响。

### 6.2 私信和消息聚合

- **P0/一期**：会话列表、历史消息分页、文字/表情/图片消息、消息合法性校验和发送者权限。
- **P0/一期**：已读、未读数、单条/全部已读、撤回时间窗、删除本地会话和服务端留痕。
- **P0/一期**：拉黑/封禁后禁止新消息；已发送内容按治理策略可见、屏蔽或保留取证。
- **P1/一期**：消息通知聚合：申请、同意/拒绝、评论、点赞、关注、活动和系统通知。
- **P1/一期**：通知偏好、分页、未读计数、全部已读和关联业务 payload。
- **P1/一期**：聊天文本敏感词实时检测、消息屏蔽/替换/人工审核、聊天举报和证据保全。
- **P2/二期**：WebSocket/实时推送、送达回执、输入状态、离线补偿和推送失败重试。
- **P2/二期**：语音、电话申请、交换微信、恋爱模式、交换答案及权限审计。

## 7. 我的模块

### 7.1 个人资料和媒体

- **P0/一期**：基础资料、昵称、头像、性别/生日不可变字段、现居地、学历、工作、婚姻、收入等校验。
- **P0/一期**：自我介绍、爱情观、择偶要求、兴趣/性格标签、资料完整度计算和推荐权重同步。
- **P0/一期**：头像、相册、封面、视频上传；文件校验、媒体审核状态、排序、删除和访问授权。
- **P0/一期**：个人主页预览、公开字段/敏感字段隐藏、对不同用户身份返回不同资料视图。
- **P1/一期**：资料修改历史、认证字段锁定、媒体审核失败原因和重新提交。

### 7.2 互动、隐私和安全

- **P0/一期**：我的申请、收藏、爆灯、浏览记录、我的动态、报名记录查询。
- **P0/一期**：隐私设置：交友限制、在线状态、公开/委托红娘/私密/暂停服务/已脱单、敏感信息隐藏。
- **P0/一期**：黑名单/屏蔽列表、解除拉黑、关系和内容查询统一应用屏蔽规则。
- **P0/一期**：消息通知偏好、隐私授权记录、用户举报记录和处理结果查询。
- **P1/一期**：安全中心、防骗指南、客服/意见反馈/FAQ/政策文本内容接口。
- **P2/二期**：任务、签到、邀请、积分账本、会员套餐、充值、积分兑换和权益核销。
- **P2/二期**：账号换绑、注销数据清理、会员权益、支付订单、退款和账务流水。

## 8. 管理后台（社区 + 消息 + 红娘）

### 8.1 内容审核和处置

- **P0/一期**：帖子、评论、图片、视频、语音、纸飞机、聊天举报的统一审核队列和筛选。
- **P0/一期**：查看原文、命中词、上下文、用户风险画像、历史违规、关联举报和证据附件。
- **P0/一期**：审核动作：通过、屏蔽、删除、替换敏感词、限制传播、限制发言、转人工、恢复。
- **P0/一期**：聊天内容按举报或风控授权查看；默认最小可见，所有查看和导出写审计日志。
- **P0/一期**：敏感词规则增删改、级别、匹配范围、替换符、版本、灰度、测试和回滚。
- **P1/一期**：批量审核、批量屏蔽/删除、审核 SLA、复审、申诉、证据保留期限和导出权限。

### 8.2 用户与红娘运营

- **P0/一期**：用户搜索、详情、认证状态、登录设备、举报记录、封禁/解封、拉黑/解除拉黑。
- **P0/一期**：封禁类型、原因、开始/结束时间、范围（登录/发帖/评论/私信/申请认识）、申诉和操作日志。
- **P0/一期**：红娘申请审核、资质审核、服务上下架、服务范围、价格展示、案例审核和投诉处理。
- **P1/一期**：红娘订单/预约/服务进度、转派、超时、评价异常和结算数据查询。
- **P1/一期**：社区话题、纸飞机规则、举报处理、推荐置顶内容和运营 Banner 管理。
- **P2/二期**：活动审核、报名管理、签到、退款审批、AI 审核复核和运营报表。

### 8.3 消息治理

- **P0/一期**：按会话/用户/时间/举报单查询消息，显示脱敏内容和消息治理状态。
- **P0/一期**：对消息执行屏蔽、删除展示、敏感词替换、限制发送、封禁账号；保留不可篡改的原始取证记录。
- **P1/一期**：系统通知、模板、定向发送、撤回、失败重试、发送范围和发送审计。
- **P2/二期**：实时推送运营、消息质量报表、自动化风控策略和批量治理。

## 9. 管理后台（首页 + 我的）

### 9.1 用户、资料和认证管理

- **P0/一期**：用户列表、条件筛选、资料详情、账号状态、资料可见状态和推荐资格。
- **P0/一期**：实名认证结果查询、认证失败处理、人工复核、认证状态修正和审核日志。
- **P1/一期**：学历/房产/婚姻/单身承诺审核、材料访问控制、过期复审和批量审核。
- **P0/一期**：头像/相册/视频审核、封面设置、违规媒体下架和重新审核。

### 9.2 首页推荐和商业化运营

- **P0/一期**：推荐候选池参数、基础筛选项、黑名单/封禁排除、推荐异常监控。
- **P1/一期**：推荐规则权重、推荐理由、每日刷新任务、人工置顶和推荐池运营。
- **P2/二期**：AI 合拍度规则配置、A/B 实验、会员专区、付费曝光套餐和效果报表。
- **P2/二期**：会员套餐、积分商品、申请次数、签到/邀请任务、支付订单、退款和账务对账。

### 9.3 配置、政策和数据报表

- **P0/一期**：协议政策、FAQ、安全指南、城市/学历/职业/标签等字典配置，版本和生效时间。
- **P1/一期**：用户注销、隐私请求、反馈、客服工单、举报申诉和处理时效报表。
- **P1/一期**：注册、认证、资料完善、推荐、申请认识、匹配、聊天、发帖和举报漏斗。
- **P2/二期**：会员收入、活动报名、红娘服务、积分发放/消耗、留存和转化报表。

## 10. 一期建议交付顺序

1. **P0：基础底座与安全**：数据模型、迁移、登录会话、RBAC、统一状态拦截、审计、限流、词库和举报模型。
2. **P0：账号与资料**：手机号/微信登录、身份选择、协议、基础资料、头像媒体、实名认证门禁。
3. **P0：核心匹配闭环**：推荐、筛选、浏览、收藏、申请认识、同意/拒绝、匹配、会话、文字消息、通知。
4. **P0：社区最小闭环**：帖子、图片、评论、点赞、关注、信息流、举报、审核和后台治理。
5. **P0/P1：红娘最小闭环**：红娘申请审核、红娘列表/主页、牵线申请、后台分配和服务状态。
6. **P0/P1：后台可运营**：用户封禁/拉黑、内容屏蔽/删除/星号替换、认证审核、敏感词配置、申诉和审计。
7. **P1：一期验收**：真实数据库集成、跨用户授权、并发幂等、封禁/拉黑传播、第三方回调和核心链路压测。

## 11. 二期建议范围

- 五重认证完整闭环和多供应商人脸服务切换。
- AI 红娘、AI 合拍度、推荐实验和精细化曝光。
- 会员、积分、支付、退款、佣金、活动报名和运营结算。
- 语音/实时聊天、实时推送、电话和微信交换。
- 纸飞机完整语音/图片玩法、情感实验室和解锁权益。
- 社区活动、话题运营、AI 审核、自动风控策略和数据报表。

## 12. 矛盾点与模糊点：已确认方案与待定事项

### 12.1 模块边界冲突

- “红娘申请”在清爽版归登录/认证，但终极版又作为牵线入口；建议：账号身份归登录模块，红娘业务资料/审核/服务归红娘模块。
- “申请认识”出现在首页、我的、消息和社区；建议：创建/扣次数归关系域，申请列表与通知归消息域，入口模块只负责调用。
- “隐私、黑名单、举报”横跨我的、社区、首页和消息；建议：建立独立安全/治理公共服务，由各模块统一调用。
- 后台按“社区+消息+红娘”和“首页+我的”拆 UI，但后端权限不应按页面拆；建议使用统一管理域和数据权限。

### 12.2 已确认的业务方案

- **实名认证是当前唯一必须建设的认证**：包含姓名、身份证号和人脸识别。学历、房产、单身承诺、婚姻认证等五重认证扩展放入二期。
- **实名认证功能门禁尚未全部定稿**：后端先提供统一认证状态和权限校验能力，具体哪些功能必须实名由产品确认后配置，不在各业务接口中硬编码。
- **认证后不可修改的资料尚未定稿**：先区分认证字段与普通资料字段；认证字段支持锁定/重新认证状态，具体字段名单由产品确认。
- **申请认识次数规则暂不固定**：次数、刷新时间、会员额度、拒绝/过期是否返还、重复申请如何计数均做成配置；申请流程必须统一执行“检查次数 -> 扣减次数 -> 创建申请 -> 发送通知”，并保证幂等和事务一致性。
- **爆灯规则已确定**：一次爆灯有效三天；不可撤回；对方不能关闭；展示在首页会员专区的个人名片中；不改变推荐排序；不支持退款。爆灯记录应包含生效时间、失效时间、展示位置和状态。
- **收藏为完全私密**：用户 A 收藏用户 B 后，仅 A 可以在“我的收藏”中查看，用户 B 和其他用户不可见，也不对外返回收藏者或收藏数量。
- **拉黑后保留历史聊天**：拉黑只限制后续关系和操作，包括不能私信、不能申请认识、不出现在推荐中、不能查看受限资料；历史聊天保留，但新消息发送必须被拦截。
- **举报必须人工审核**：举报创建举报单并进入人工审核队列，不默认等同于立即封禁；审核结果、处理人、处理时间、证据和申诉记录必须留存。
- **封禁支持分级限制**：至少支持总封禁、限制发帖、限制评论、限制私信、限制申请认识等类型；每种限制有原因、开始时间、结束时间、操作人和审计记录。
- **注销采用软删除/匿名化方案**：帖子显示为已注销用户；历史聊天保留必要取证但隐藏用户身份；认证材料按合规期限删除；举报记录按审计期限保留。
- **敏感词处理方案保留三种结果**：高风险内容直接拦截；一般违规内容按规则替换为 `*`；复杂或争议内容转人工审核。后台只有一种管理员身份，即超级管理员，拥有全部后台权限；查看聊天原文、导出证据等敏感操作必须记录审计日志。
- **纸飞机仅对用户侧匿名**：平台和超级管理员可以看到真实用户关联关系，风控和治理可以追溯；用户侧展示头像/昵称与真实资料分离。具体“解锁”资料范围和积分消耗属于二期商业化规则。
- **人脸认证第三方接入属于二期**：一期只确定实名认证领域模型、状态机和接口适配边界，不接入真实供应商，不把人脸服务商字段写死在核心业务中。
- **会员、积分、支付、付费曝光、活动、私人定制和红娘付费咨询属于二期**：一期只保留必要的业务扩展字段和接口边界，不在一期实现支付订单和财务结算。
- **管理员角色统一为超级管理员**：不再拆分内容审核员、客服、风控管理员等角色；但接口仍需保留菜单权限、数据范围、敏感数据查看、导出和审计能力，便于后续扩展而不破坏接口。

### 12.3 现有矛盾和模糊点

- **哪些功能必须实名认证仍未确定**：`1.txt` 已确认实名认证是必需认证，但没有确认浏览、收藏、点赞、关注、发帖、评论、申请认识、聊天和纸飞机分别需要什么条件。建议产品输出“功能 - 认证状态 - 限制结果”矩阵。
- **哪些资料属于认证字段仍未确定**：性别、生日、婚姻状态、学历、房产、收入、工作等资料是否认证后不可修改，尚未形成字段清单；否则无法冻结资料表、修改接口和审核状态机。
- **申请认识次数规则仍未确定**：需要确认普通用户次数、刷新时刻、是否累计、拒绝/过期是否返还、重复申请是否扣次、会员和积分规则。当前只能先做配置化，不能把数字写死在代码中。
- **“双重认证”产品术语仍与当前方案冲突**：当前确认只有实名认证是必须认证，但终极版仍多次出现“双重认证”。需要产品决定废弃该术语，或明确它代表实名认证加哪一种二期认证。
- **实名认证与人脸服务的分期存在冲突**：`1.txt` 已确定实名认证包含姓名、身份证号和人脸识别，但真实人脸第三方接入又确定为二期。建议一期只完成认证领域模型、申请/查询/状态机、供应商适配接口和测试环境 Mock/人工审核；一期生产环境不得把“已完成实名认证”作为真实人脸已验真的结论，依赖实名的生产功能应等二期人脸服务上线后开放。
- **举报人工审核后的默认处置未确定**：举报本身不等于封禁，但审核通过后是屏蔽内容、限制账号、总封禁还是仅记录，需要按举报类型定义处置策略。
- **封禁对历史内容的处理未完全确定**：已确定分级限制，但还需明确总封禁或限制发言后，历史帖子、评论、聊天是否继续对其他用户可见，是否需要批量隐藏。
- **敏感词证据留存期限未确定**：已确定三种处理方式和超级管理员权限，但原始内容、命中记录、替换内容、人工审核记录保存多久仍需合规确认。
- **爆灯的触发方式和权益来源仍未确定**：已确定有效三天、不可撤回、不退款、不影响排序，但还需确定一期是否开放免费爆灯、由什么入口触发，以及是否需要通知对方。
- **纸飞机解锁规则仍未确定**：平台可追溯已确定，但解锁哪些资料、是否需要对方同意、积分消耗、重复解锁、解锁后的资料有效期仍待二期产品确认。
- **推荐规则和隐私展示仍未确定**：推荐权重、刷新时刻、距离精度、IP 展示粒度、收入/年薪是否展示、访客信息是否脱敏仍需确认。
- **外部服务范围和生产参数仍未确定**：短信、微信登录、对象存储、内容审核、地图/LBS 属于一期依赖，供应商、回调、密钥、超时、重试和降级方案仍需确认；人脸和支付明确放二期。

### 12.4 外部依赖与验收前置条件

- 短信、微信登录、对象存储、内容审核和地图/LBS 属于一期依赖；人脸认证、支付及会员/积分相关服务属于二期依赖，不能混入一期验收范围。
- 不能以当前 Mock 测试替代真实第三方验收；一期必须覆盖短信/微信/存储/内容审核服务的超时、重复回调、验签失败、供应商拒绝和服务不可用。
- 需要产品补充功能认证门禁、资料锁定字段、申请次数、举报处置、历史内容可见性、敏感证据留存期限和推荐展示规则后，才能冻结一期 API 契约。
- 管理员角色当前只定义超级管理员；仍需确定超级管理员内部是否需要操作审计、敏感数据查看确认和导出审批等二次控制。
- 所有新接口、字段、权限和状态变更必须同步 `docs/api/`，并按项目规则补充测试、审计和敏感配置说明。

### 12.5 建议的落地顺序

1. 先冻结实名认证状态和功能门禁矩阵；一期实现姓名、身份证号、认证状态机、供应商适配接口和 Mock/人工审核流程，真实人脸第三方接入及生产实名闭环放二期。
2. 冻结资料字段分类：认证后锁定、修改后重新审核、普通可修改三类。
3. 实现统一申请认识服务，次数规则使用配置表，保证检查、扣减、创建申请和通知的一致性。
4. 实现统一安全服务：拉黑限制但保留历史聊天；举报进入人工审核；封禁支持总封禁和按功能限制。
5. 实现敏感词规则和内容状态机；超级管理员可以治理内容，但原文查看和证据导出必须审计。
6. 实现爆灯一期固定规则：三天有效、不可撤回、不可关闭、不影响排序、不退款，并预留二期商业化字段。
7. 将会员、积分、支付、活动、纸飞机解锁和人脸供应商接入全部列入二期需求冻结，不作为一期完成条件。

## 13. 一期完成判定

- 新用户可以安全登录、接受协议、完成必要资料和实名认证，并在权限不足时得到明确状态。
- 用户可以获取合规推荐、查看资料、收藏、申请认识；对方同意后只能建立一个有效匹配和会话。
- 匹配用户可以收发文字消息；拉黑、封禁、举报和敏感词规则即时影响后续操作。
- 用户可以发布和浏览社区内容；帖子、评论、聊天均有审核、举报、屏蔽、删除和审计闭环。
- 管理员可以按权限完成用户封禁/解封/拉黑、内容治理、认证审核、红娘审核和敏感词配置。
- 关键状态变更具备幂等、事务一致性、审计记录、异常告警和真实 MySQL/Redis 集成测试。
