# 婚恋小程序功能清单及遗漏评估结果

> 评估范围：用户与认证模块、社交互动模块、个人中心模块
>
> 评估依据：一期/二期功能规划，以及更新后的《婚恋小程序功能规格书》
>
> 评估日期：2026-07-18

## 一、评估说明

本文仅评估产品功能清单，不评价后端代码实现、技术架构或数据库设计。

功能状态说明：

- **已覆盖**：更新后的功能规格书中已有对应功能。
- **需补充**：原始一期或二期规划要求，但更新后的功能规格书没有明确覆盖。
- **不完整**：功能已经出现，但缺少关键流程、状态、权限或异常规则。
- **矛盾**：不同功能之间存在规则冲突，需要统一后再进入开发。

## 二、总体结论

更新后的功能规格书已经覆盖了大部分一期和二期功能，但还不能直接作为最终开发验收标准。

当前最重要的问题是：

1. **双向喜欢匹配没有形成完整闭环。**
2. **喜欢、收藏、关注三种关系定义混淆。**
3. **个人中心缺少“我的喜欢”和“我的匹配”。**
4. **实名认证没有明确姓名 + 身份证号二要素校验。**
5. **资料完整度算法合计超过 100%。**
6. **申请次数扣减、退还和过期规则没有统一。**
7. **浏览次数和访客查看规则存在不一致。**

# 第一期：MVP 核心闭环

## 一、用户与认证模块

### 1.1 登录注册

#### 具体功能点

- 微信一键登录。
- 调用微信 `wx.login()` 获取登录凭证。
- 后端通过 code 换取 openid。
- 首次登录自动创建用户。
- 已注册用户直接登录。
- 自动获取微信昵称和头像。
- 手机号验证码登录。
- 输入手机号并发送短信验证码。
- 未注册手机号自动注册。
- 手机号唯一性校验。
- 微信账号与手机号绑定。
- 微信登录后检测是否已绑定手机号。
- 未绑定手机号时强制引导绑定。
- 退出登录。
- 服务端使 Token 失效。
- 本地清除登录凭证。

#### 当前覆盖情况

基本已覆盖微信登录、手机号登录、绑定手机号和退出登录。

#### 遗漏或不完整

- 没有明确 Token 刷新机制。
- 没有明确多设备登录策略。
- 没有明确异地登录和异常登录提醒。
- 没有明确手机号换绑流程。
- 没有明确微信账号解绑规则。
- “绑定手机号后可使用手机号作为账号密码登录”，但没有说明首次如何设置密码。
- 没有明确是否必须设置密码。
- 没有明确忘记密码和修改密码流程。
- 没有明确手机号已绑定其他微信账号时的处理方式。

### 1.2 短信验证码

#### 具体功能点

- 验证码有效期 5 分钟。
- 同一手机号 60 秒内不能重复发送。
- 同一手机号 24 小时最多发送 10 次。
- 验证码错误 3 次后锁定 15 分钟。
- 验证码使用后立即失效。
- 验证码不可重复使用。
- 短信发送失败后的重试提示。
- 防止恶意批量发送短信。

#### 当前覆盖情况

手机号登录中已经覆盖验证码有效期、发送频率和错误锁定规则。

#### 遗漏或不完整

- 没有明确 IP、设备、手机号三种维度的限流规则。
- 没有明确短信发送失败是否计入次数。
- 没有明确连续请求新验证码时旧验证码是否失效。
- 没有明确验证码过期后的错误提示。
- 没有明确短信服务异常时的备用处理方案。

### 1.3 安全承诺与协议签署

#### 具体功能点

- 注册后展示《交友安全自律承诺书》。
- 用户确认本人为单身状态。
- 用户确认文明交友。
- 用户勾选确认后才能进入资料完善流程。
- 记录签署时间。
- 记录用户同意的协议版本。
- 后台可修改承诺书内容。
- 协议更新后重新要求用户确认。

#### 当前覆盖情况

安全承诺签署已经覆盖。

#### 遗漏或不完整

- 没有明确是否同时签署用户服务协议、隐私政策和社区行为规范。
- 没有明确未成年人使用限制协议。
- 没有明确协议版本管理方式。
- 没有明确用户拒绝协议后的处理方式。
- 没有明确协议更新后是否阻断核心功能。

### 1.4 实名认证

#### 具体功能点

- 姓名填写。
- 身份证号填写。
- 姓名 + 身份证号二要素校验。
- 对接第三方实名认证接口。
- 认证状态包括未认证、认证中、已认证、认证失败。
- 展示认证失败原因。
- 认证失败后重新提交。
- 身份证信息脱敏展示。
- 实名认证成功后展示认证标识。
- 只有实名认证用户才能申请认识或聊天。

#### 当前覆盖情况

规格书包含实名认证和人工审核，但主要描述为上传身份证资料，未完整定义二要素校验流程。

#### 遗漏或不完整

- 没有明确姓名 + 身份证号二要素校验。
- 没有明确第三方实名认证接口。
- 没有明确认证失败后的重试流程。
- 没有明确认证审核中的状态。
- 没有明确认证资料是否允许修改。
- 没有明确认证通过后修改姓名、身份证号的限制。
- 没有明确实名认证是否是浏览、申请、聊天的强制门槛。

### 1.6 头像与相册

#### 具体功能点

- 拍照或从相册选择。
- 头像裁剪。

#### 具体功能点

- 学历下限。
- 收入下限。
- 婚姻状况。
- 籍贯省份。

#### 遗漏或不完整

- 理想省份。

### 1.8 资料完整度

#### 具体功能点

- 登录后提示完善资料。
- 展示 0~100% 完整度。
- 展示未完成项目。
- 每完成一项实时增加进度。
- 达到指定比例后允许浏览。
- 达到 100% 后允许推荐、申请等核心功能。

#### 矛盾与不完整

当前权重合计超过 100%：

- 基础资料：30%。
- 详细资料：32%。
- 头像：15%。
- 自我介绍：10%。
- 相册：10%。
- 兴趣标签：5%。
- 认证项至少：15%。

合计至少为 117%，需要重新设计权重，或改为“完成项数量 / 总完成项数量”。

还需明确：

- 达到多少比例可以进入首页。
- 达到多少比例可以申请认识。
- 实名认证是否属于完整度必选项。
- 未认证用户是否可以浏览他人。
- 删除资料后完整度是否下降。

### 1.9 用户与认证模块一期问题解决方案

本节针对 1.1 至 1.8 中列出的遗漏、不完整和矛盾逐项给出一期建议方案。以下方案优先保证“注册可用、身份可信、资料可控、核心交友功能有明确门槛”。

#### 1.9.1 登录、Token 和账号关联方案

**推荐规则：**

- 微信登录和手机号验证码登录都使用同一个内部用户 ID，不能因为登录方式不同创建两个用户。
- 微信登录只负责确认微信身份；手机号绑定后才视为完成手机号认证。
- 微信登录成功后返回 `need_bind_phone` 状态。未绑定手机号的用户只能访问登录、协议、绑定手机号和基础资料引导页面，不能进入推荐、申请、聊天和发布互动内容。
- Access Token 有效期建议为 2 小时，Refresh Token 有效期建议为 30 天。Refresh Token 每次刷新后轮换，旧 Refresh Token 立即失效。
- 服务端保存 Refresh Token 的哈希值、用户 ID、设备 ID、创建时间、过期时间、撤销时间和最后使用时间。
- 退出登录时撤销当前设备会话；用户执行“退出所有设备”时撤销该账号全部会话。
- Token 中只保存用户 ID、会话 ID、Token 类型和过期时间，不保存手机号、身份证号等敏感信息。

**多设备和异常登录：**

- 一期允许同一账号最多 5 个有效设备会话，超过数量时提示用户管理设备。
- 登录日志记录登录方式、时间、设备、IP 和地区，但手机号、Token 等敏感字段不得写入日志。
- 新设备、异常地区或短时间内频繁登录时，向用户发送站内安全提醒。
- 提供“登录设备管理”入口，用户可以查看并移除其他设备。

**手机号换绑和微信解绑：**

- 换绑手机号必须验证当前登录状态，并验证旧手机号或完成实名二次校验。
- 新手机号必须先校验验证码，再检查是否已经绑定其他账号。
- 一个手机号只能绑定一个账号；如果已绑定其他账号，不允许直接覆盖绑定。
- 一期不建议开放微信解绑。若必须开放，需确保账号至少保留手机号或其他可用登录方式，避免账号失去登录入口。

**密码策略：**

- 一期建议只使用手机号验证码登录，不上线密码登录，避免同时维护两套认证流程。
- 如果产品确定需要密码登录，则必须增加“设置密码、修改密码、忘记密码、连续错误锁定”四个完整流程。
- 绑定手机号不应自动等同于设置密码；产品文案和实际登录方式必须保持一致。

**验收标准：**

- 同一个微信账号和手机号登录后始终进入同一个用户账号。
- 退出登录后旧 Access Token 和 Refresh Token 均不能继续换取有效会话。
- 未绑定手机号的用户无法进入申请和聊天页面。
- 已绑定手机号不能被其他账号重复绑定。

#### 1.9.2 短信验证码方案

**限流和生命周期规则：**

- 手机号维度：同一手机号 60 秒内最多发送 1 次，24 小时最多发送 10 次。
- IP 维度：同一 IP 每小时设置发送上限，超过后进入人工验证或临时封禁。
- 设备维度：同一设备短时间内绑定多个手机号时触发风控。
- 验证码发送、校验、绑定手机号和登录接口都必须经过限流。
- 验证码有效期为 5 分钟。
- 同一手机号再次发送验证码时，旧验证码立即失效，只允许最新验证码通过。
- 验证码校验成功后立即失效。
- 连续错误 3 次后锁定该手机号 15 分钟。
- 发送失败不消耗有效验证码次数；第三方确认发送成功时才计入发送次数。

**异常处理：**

- 手机号格式错误、发送过于频繁、验证码过期、验证码错误、账号锁定分别返回不同提示。
- 前端倒计时只用于展示，不能作为服务端安全依据。
- 短信服务不可用时只提示稍后重试，不展示第三方内部错误。
- 登录验证码和换绑验证码必须区分用途，避免登录验证码被用于换绑手机号。

**验收标准：**

- 旧验证码不能在新验证码发送后继续使用。
- 验证码不能重复使用。
- 3 次错误后确实锁定 15 分钟。
- 同一 IP、设备和手机号的异常批量请求可以被限制。

#### 1.9.3 协议和安全承诺方案

**协议清单：**

- 《用户服务协议》
- 《隐私政策》
- 《交友安全自律承诺书》
- 《社区行为规范》
- 《未成年人保护与使用限制说明》

**签署规则：**

- 注册完成后必须确认用户服务协议和隐私政策。
- 进入资料完善前必须确认交友安全承诺书。
- 进入社区、纸飞机和聊天前必须确认社区行为规范。
- 用户拒绝协议时，保留登录退出能力，但禁止进入需要该协议授权的功能。
- 协议确认记录保存用户 ID、协议编码、版本号、确认时间和确认场景。
- 协议内容修改后生成新版本，不覆盖旧版本。
- 重大隐私或安全条款变更后，用户再次登录时必须重新确认。

**后台管理：**

- 协议支持草稿、发布、下线和历史版本查询。
- 已发布版本不能直接修改，只能生成新版本。
- 后台记录发布人、发布时间和变更说明。

**验收标准：**

- 新用户未完成必要协议确认时不能使用对应核心功能。
- 可以查询任一用户在指定时间同意的协议版本。
- 协议更新后旧版本仍可追溯。

#### 1.9.4 实名认证方案

**一期认证流程：**

1. 用户提交姓名和身份证号。
2. 前端完成格式校验，后端再次校验姓名长度、身份证号格式和年龄。
3. 后端调用第三方二要素实名认证服务。
4. 成功后将认证状态更新为“已认证”。
5. 失败后记录失败原因，但前端只展示安全、可理解的提示。
6. 多次失败后进入人工复核或暂时限制重试。

**认证状态：**

- 未认证。
- 已提交，等待校验。
- 已认证。
- 认证失败。
- 人工复核。
- 认证撤销。

**数据和权限规则：**

- 身份证号不能明文写入日志或接口响应。
- 前端只展示脱敏后的认证信息。
- 原始身份证号如确需保存，必须加密保存，并限制授权管理员访问。
- 实名认证通过后，用户不能自行修改姓名和身份证号。
- 年龄必须满足 18 周岁要求，出生日期与身份证信息不一致时认证失败。
- 未实名认证用户可以登录、完善资料和浏览公开内容，但不能发起申请或聊天。
- 认证失败或被撤销时，立即停止新的申请和聊天权限，保留申诉和重新认证入口。

**验收标准：**

- 姓名和身份证号不匹配时不能认证成功。
- 认证中状态下不能重复创建多个认证任务。
- 认证成功后不能直接修改认证身份信息。
- 实名状态变更能够同步影响申请和聊天权限。

#### 1.9.5 基础资料和成年人校验方案

**字段规则：**

- 出生日期为必填项，服务端计算年龄，前端不能直接提交年龄作为可信数据。
- 年龄必须达到 18 周岁；未成年人不能注册或使用交友功能。
- 性别一期允许选择一次，提交后不能自行修改；确需修改时进入人工审核。
- 身高限制为 150~200cm，超出范围拒绝保存。（去除身高限制）
- 职业、学历、婚姻状况使用预设枚举，“其他”才允许补充文字。
- 月收入使用收入区间，避免要求用户填写精确收入。
- 籍贯和工作地统一使用省、市、区编码。
- 昵称设置长度、敏感词和修改频率限制。

**资料修改规则：**

- 普通资料修改后立即生效；性别、出生日期和实名信息修改必须审核。
- 被封禁用户禁止修改对外资料，但可以查看账号状态和提交申诉。
- 资料被举报或审核失败时，只标记具体字段和原因，不直接清空全部资料。
- 保存资料时执行类型、长度、枚举和业务范围校验。

**验收标准：**

- 未成年人不能进入推荐、申请和聊天功能。
- 出生日期修改不会直接改变已认证身份年龄。
- 非法枚举值、超范围身高和超长文本不能保存。（去除身高限制）
- 地区字段在推荐和筛选中使用同一套编码。

#### 1.9.6 头像、相册和视频方案

**文件规则：**

- 头像支持 JPG、PNG，单张不超过 5MB。
- 相册图片建议单张不超过 10MB，最多 9 张。
- 视频限制为 MP4，时长不超过 30 秒，并设置文件大小上限。
- 服务端根据真实 MIME 类型校验，不能只信任文件扩展名。
- 图片上传后生成原图、压缩图和缩略图；对外展示默认使用压缩图。
- 相册保存排序号和首图标记；删除首图后自动将第一张有效图片设为首图。

**审核流程：**

- 上传后状态为“审核中”，审核通过后公开展示。
- 审核失败时保存失败原因，用户可以删除或重新上传。
- 色情、广告、二维码、他人肖像和违法内容不得公开。
- 头像被举报后进入复核，不应仅凭一次举报永久删除。
- 高风险文件可以先隐藏展示，再进入人工复核。

**存储和删除：**

- 文件路径包含用户 ID 和随机文件 ID，禁止使用可猜测的连续文件名。
- 注销账号进入冷静期时停止公开访问；冷静期结束后删除原图、压缩图、缩略图和视频。
- 删除文件时同时清理数据库记录和对象存储文件。
- 文件访问使用受控 URL，不能通过公开目录绕过隐私设置。

**验收标准：**

- 修改扩展名不能绕过真实文件类型校验。
- 超出格式、大小和时长限制的文件不能上传。
- 审核失败内容不会出现在推荐卡片和个人主页。
- 首图删除后仍然存在有效首图。

#### 1.9.7 择偶要求和城市方案

**字段设计：**

- 期望年龄最小值和最大值。
- 期望身高最小值和最大值。
- 最低学历。
- 最低收入区间。
- 婚姻状况要求。
- 期望省份。
- 期望城市，可多选。
- 是否接受异地。
- 是否接受跨省。
- 是否有房要求。
- 是否抽烟、饮酒要求。
- 其他文字要求。

**默认值和校验：**

- 未填写条件默认为“不限”，不能默认为严格条件。
- 年龄和身高区间必须满足最小值不大于最大值。
- 城市条件为空时不限制城市。
- “仅同城”必须明确使用工作地还是居住地。
- 省市区使用统一地区编码，不允许只提交中文名称作为唯一依据。

**推荐影响：**

- 修改择偶要求只影响后续推荐，不回溯修改历史推荐名单。
- 硬性条件用于过滤，软性条件用于排序和合拍度计算。
- 结果不足时提示已放宽的条件，并允许用户恢复原条件。

**验收标准：**

- 保存后重新进入页面，所有择偶条件保持一致。
- 修改城市后新推荐使用新城市条件。
- 空条件不会错误过滤全部用户。
- 城市筛选、推荐算法和个人主页展示使用一致的城市定义。

#### 1.9.8 资料完整度方案

**建议采用“总权重 100% + 权限门槛”方案：**

| 资料项 | 建议权重 |
|---|---:|
| 性别、出生日期、地区 | 20% |
| 婚姻状况 | 8% |
| 职业、学历、收入 | 15% |
| 身高 | 5% |
| 头像 | 15% |
| 自我介绍 | 10% |
| 相册 | 10% |
| 兴趣和性格标签 | 7% |
| 择偶要求 | 5% |
| 实名认证 | 5% |
| **合计** | **100%** |

**权限门槛：**

- 0%~39%：只能访问登录、协议和资料引导页面。
- 40%~59%：可以浏览首页推荐，但不能申请认识。
- 60%~79%：可以浏览完整资料并使用喜欢、收藏等基础互动。
- 80%~99%：可以申请认识，但必须完成实名认证。
- 100%：资料达到完整状态，可以使用一期完整交友流程。

**计算规则：**

- 每个资料项只能计算一次，不能因同时满足多个条件重复加分。
- 删除或清空资料后实时扣除对应分值。
- 认证项只有认证成功后计分，提交中不计为完成。
- 相册至少有 1 张有效图片才计入相册分值。
- 兴趣标签至少达到 3 个才计入标签分值。
- 自我介绍达到最低有效字数并通过内容审核后计入分值。
- 最终分值由服务端计算，前端只负责展示。

**验收标准：**

- 所有资料项满分合计必须为 100%。
- 新增、修改、删除资料后分值能够重新计算。
- 未实名认证用户不能通过填写其他资料绕过申请门槛。
- 前端显示的完整度与服务端权限判断一致。

当前规格书把“喜欢”和“收藏”写成同一行为：

## 5.2 思维导图新增或明确的产品结构

### 全局导航结构

思维导图将产品明确为 5 个一级 Tab：

5. 我的：个人资料、互动记录、认证、任务、海报、报名、充值、隐私、客服和协议。

### 注册身份分流

注册登录之后新增明确的身份选择：

- 自己找。
- 父母帮找。
- 找搭子。
- 红娘招聘分支：推广红娘、合伙人招募、服务红娘。

该分流不是普通的“交友状态”，而是会影响用户后续看到的首页、匹配规则、权限、协议和功能入口，应该作为注册业务数据保存并进入权限判断。

## 5.3 现有后端覆盖情况

| 思维导图能力 | 当前后端状态 | 评估结论 |
|---|---|---|
| 微信登录、手机号登录、绑定手机号 | 已有 `/api/v1/auth/*` 接口 | 部分覆盖 |
| 协议签署、实名认证、基础资料 | 已有认证和资料接口 | 部分覆盖 |
| 注册身份选择 | 没有请求字段、数据字段或接口 | 未覆盖 |
| 自己找 / 父母帮找 / 找搭子 | 没有业务模型和权限判断 | 未覆盖 |
| 推广红娘 | 没有路由和申请类型 | 未覆盖 |
| 合伙人招募 | 没有路由和申请类型 | 未覆盖 |
| 服务红娘招聘 | 数据库有通用 `user_matchmaker_apply` 表，但没有 API 和审核服务 | 仅数据库预留 |
| 首页推荐和广场名片流 | 没有推荐、广场、筛选接口 | 未覆盖 |
| 社区动态、话题、点赞、评论、关注 | 数据库有部分表，但没有 API | 仅数据库预留 |

| 认识申请和聊天 | 数据库有 `match_apply`、`chat_session`、`chat_message`，但没有 API | 仅数据库预留 |

当前实际可调用的业务接口仍主要集中在：认证、会话、协议、实名提交和个人资料。思维导图所描述的完整产品主路径尚未在 FastAPI 路由层形成闭环。

### 已有可复用表

数据库脚本已经预留了以下能力：

- `user_favorite`：喜欢、收藏、关注三种关系。
- `user_match`：匹配关系。
- `user_match_recommend`、`user_swipe_record`：推荐和滑动行为。
- `match_apply`：认识申请。
- `chat_session`、`chat_message`：聊天。
- `community_post`：社区动态。
- `user_matchmaker_apply`：红娘申请。
- `matchmaker_service`、`matchmaker_rating`：红娘服务和评价。
- `user_membership`、`user_points`、`user_task`：会员、积分和任务。
- `activity` 相关表：线下活动和报名。

### 当前结构不足

1. `users` 没有注册身份字段，无法区分“自己找、父母帮找、找搭子”。
2. `user_matchmaker_apply` 没有明确的申请分支字段，无法区分推广红娘、合伙人和服务红娘。
3. 现有红娘申请表只有统一审核状态，缺少不同身份的资质、合同、分成、服务范围和审核规则。
4. 没有明确普通用户身份与红娘身份是否可以同时存在，也没有角色切换和撤销规则。
5. 没有角色权限模型，无法限制服务红娘、推广红娘、合伙人、普通用户和管理员的接口权限。
6. `user_privacy.match_status` 表示交友展示状态，不能替代注册身份选择字段。
7. “找搭子”与婚恋匹配的年龄、性别、隐私、聊天和推荐规则尚未定义，不能直接复用婚恋规则。

## 5.5 对现有后端代码的评估

### 结论：需要调整，但当前不建议直接修改现有认证接口

现有认证接口可以继续作为统一登录入口，但在登录后的注册引导阶段需要增加身份选择流程。建议保持“认证身份”和“业务角色”分离：

- 认证身份：微信、手机号、实名状态、会话状态。
- 用户意图：自己找、父母帮找、找搭子。
- 平台角色：普通用户、推广红娘、合伙人、服务红娘、管理员。
- 申请状态：待提交、审核中、通过、驳回、暂停、撤销。

不建议把这些值直接混入 `users.status` 或 `user_privacy.match_status`，否则会与账号冻结、注销、暂停展示等状态发生冲突。

### 建议新增的后端能力

#### P0：注册身份分流

- 查询注册身份选项。
- 提交当前用户的注册身份。
- 查询当前用户身份及引导状态。
- 防止未完成身份选择的用户进入不适用的首页和匹配流程。
- 记录身份选择时间、来源、版本和变更记录。

建议身份枚举至少包括：`self_match`、`parent_match`、`companion`。

#### P0：红娘招聘分支

- 查询红娘招聘类型。
- 提交推广红娘申请。
- 提交合伙人招募申请。
- 提交服务红娘申请。
- 查询申请详情和审核状态。
- 撤回或重新提交被驳回申请。
- 管理端审核、驳回、暂停和恢复申请。

三类申请应有独立的 `application_type`，不能只依赖 `user_matchmaker_apply.status` 区分。

#### P1：角色权限和业务入口

- 根据注册意图返回不同首页配置和功能入口。
- 对红娘相关接口增加角色权限校验。
- 限制普通用户不能调用红娘服务管理接口。
- 限制推广红娘、合伙人和服务红娘只能访问其授权范围。
- 为角色变更、审核和撤销保留审计记录。

#### P1：五个 Tab 对应的业务接口

需要分别建设首页推荐、社区、牵线、消息和个人中心接口聚合层。现有数据库表可以作为部分基础，但不能因为已有表就视为功能已实现；目前缺少参数校验、权限校验、状态流转、分页、通知和错误码契约。

## 5.6 需要在产品层先确认的问题

在后端设计前，建议先确定以下规则：

1. 注册身份是否只能选择一个，还是允许同时选择“自己找”和“找搭子”。
2. “父母帮找”是父母独立账号，还是由子女账号授权父母代操作。
3. 父母帮找是否需要单独的监护/授权协议和隐私范围。
4. “找搭子”是否可以跳过婚姻状态、单身承诺和婚恋实名认证门槛。
5. 注册身份是否允许修改，修改是否需要冷却期或人工审核。
6. 普通用户是否可以申请成为红娘，是否要求先完成实名认证。
7. 推广红娘、合伙人和服务红娘是否允许同时申请。
8. 服务红娘是否需要职业资质、服务协议、收费规则和退款规则。
9. 推广红娘和合伙人的佣金、邀请、结算和提现规则是什么。
10. 红娘身份审核通过后，是否仍可以使用普通用户的匹配、聊天和社区功能。

## 5.7 更新后的优先级建议

### P0：进入下一轮后端设计前必须明确

1. 注册身份数据模型和身份选择接口。
2. “自己找、父母帮找、找搭子”的权限和匹配差异。
3. 红娘招聘三类申请的枚举、字段、审核状态和角色权限。
4. 父母帮找的授权关系与隐私边界。
5. 红娘身份与普通用户身份的并存、切换和撤销规则。

### P1：产品主路径后端接口

1. 首页推荐、广场、筛选和浏览记录。
2. 喜欢、收藏、关注、爆灯和双向匹配。
3. 认识申请、消息中心和聊天。
4. 社区动态、同城、话题和纸飞机。
5. 牵线、服务红娘、AI 红娘和私人定制。

### P2：商业化与运营能力

1. 会员充值、申请次数、积分和订单。
2. 任务、邀请、海报和推广关系。
3. 红娘分成、结算、评价和投诉。
4. 活动报名、客服、安全中心和管理后台。

## 5.8 最终评估结论

《宣誓爱基本功能架构（清爽版）》与现有后端相比，属于产品范围和角色模型的扩展，不是单纯增加几个页面。现有认证代码可以保留，但需要在认证成功后的资料引导阶段增加注册身份分流，并新增用户意图、红娘申请类型、角色权限和审核状态模型。

现阶段不建议直接进入社区、牵线和红娘服务接口编码；应先完成注册身份和角色权限的产品确认，否则后续推荐、聊天、订单、分成和后台审核都会产生兼容性返工。
