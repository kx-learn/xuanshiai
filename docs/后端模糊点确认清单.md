# 宣誓爱后端模糊点确认清单

## 使用说明

本清单用于产品、运营、后端和合规人员在一期开发前逐项确认。每项必须形成明确结论，不能只确认“后续再看”。确认结果会影响数据库字段、状态机、接口权限、前端提示、后台操作和测试用例。

建议状态：

- **待确认**：产品规则尚未确定，禁止在代码中写死。
- **建议采用**：推荐的一期默认方案，产品确认后即可落地。
- **已确认**：根据当前需求和已有注释已经确定。
- **二期确认**：一期不实现，但必须预留边界和数据字段。

---

## 1. 实名认证功能门禁

### 当前结论

- 当前唯一必须建设的认证是实名认证。
- 实名认证包含姓名、身份证号和人脸识别。
- 真实人脸第三方服务接入属于二期。
- 目前还不明确哪些功能必须通过实名认证。

### 必须确认

请逐项确认以下功能的最低准入条件：

| 功能 | 未登录 | 已登录未实名 | 已实名 | 备注 |
| --- | --- | --- | --- | --- |
| 浏览首页推荐 |  |  |  | 是否允许未实名浏览 |
| 查看他人主页 |  |  |  | 返回哪些资料字段 |
| 收藏用户 |  |  |  | 收藏是否要求实名 |
| 点赞/关注 |  |  |  | 是否要求实名 |
| 发布帖子 |  |  |  | 是否需要实名或人工审核 |
| 评论/回复 |  |  |  | 是否比发帖要求更高 |
| 申请认识 |  |  |  | 核心关系功能的准入条件 |
| 私信聊天 |  |  |  | 是否只要求双方匹配 |
| 发送纸飞机 |  |  |  | 是否需要实名 |
| 参加线下活动 |  |  |  | 二期能力 |
| 红娘申请 |  |  |  | 是否必须实名 |

### 建议采用

- 普通浏览、查看公开资料、收藏可以允许已登录用户使用。
- 发帖、评论、点赞、关注、申请认识、发送纸飞机建议要求实名认证，具体由产品确认。
- 私信只允许匹配成功的用户使用，同时发送方账号必须未被限制私信。
- 人脸服务未上线前，不得把 Mock 或人工测试结果当作生产环境真实实名结果。

### 后端影响

- 建立统一认证状态：`UNVERIFIED`、`SUBMITTED`、`REVIEWING`、`VERIFIED`、`REJECTED`、`EXPIRED`。
- 建立功能门禁配置，不在帖子、评论、申请认识等接口中分别硬编码判断。
- 每次拒绝都返回统一错误码和认证引导信息。

### 确认后产物

- 《功能认证门禁矩阵》
- 认证状态机
- 认证错误码和前端跳转规则

## 2. 认证后不可修改的资料

### 当前问题

目前未确定以下字段是否属于认证字段：

- 性别
- 出生日期
- 婚姻状态
- 姓名
- 身份证号
- 学历
- 房产
- 收入
- 工作和公司

### 必须确认

每个字段需要明确以下规则：

1. 是否首次填写后可以修改。
2. 是否实名认证通过后锁定。
3. 修改是否需要重新提交认证材料。
4. 修改期间旧值是否继续展示。
5. 修改是否触发推荐重新计算。
6. 是否保存修改历史。

### 建议分类

#### A 类：实名认证绑定字段

建议包括：

- 姓名
- 身份证号
- 出生日期
- 性别

实名认证成功后锁定。确需修改时，进入人工申诉或重新认证流程。

#### B 类：变更后需要重新审核字段

建议包括：

- 婚姻状态
- 学历
- 房产信息
- 认证相关职业信息

修改后进入 `REVIEWING`，审核期间可以继续展示旧值，审核通过后替换新值。

#### C 类：普通可修改字段

建议包括：

- 自我介绍
- 兴趣标签
- 择偶要求
- 普通工作描述
- 相册和视频

但仍需经过内容审核和格式校验。

### 后端影响

- 资料字段需要记录 `verified_at`、`verification_status`、`last_reviewed_at`。
- 认证字段不能只靠前端禁用编辑，后端必须拒绝非法修改。
- 需要资料修改历史表和重新审核记录。

### 确认后产物

- 《用户资料字段分级表》
- 《资料修改与重新审核流程》
- 字段锁定和错误码定义

## 3. 申请认识次数和额度规则

### 当前问题

以下规则尚未确定：

- 普通用户每天几次。
- 会员每天几次。
- 是否按自然日刷新。
- 次数是否可以累积。
- 拒绝后是否返还。
- 过期后是否返还。
- 用户主动撤回是否返还。
- 重复申请是否扣次数。
- 申请认识是否可以消耗积分。

### 必须确认表

| 规则 | 待确认内容 |
| --- | --- |
| 普通用户额度 | 例如每天 3 次 |
| 会员用户额度 | 例如每天 7 次 |
| 刷新时间 | 每日 00:00、注册时间滚动刷新或其他 |
| 是否累计 | 未使用次数是否带到下一天 |
| 拒绝处理 | 是否返还申请次数 |
| 过期处理 | 对方长期不处理是否返还 |
| 主动撤回 | 撤回是否返还 |
| 重复申请 | 存在未结束申请时是否禁止再次申请 |
| 匹配成功 | 是否立即结束其他未处理申请 |
| 会员变化 | 购买/到期时额度如何变化 |

### 建议采用

- 使用独立额度账本，不直接修改用户表中的一个数字。
- 一期将次数、刷新时间和返还规则做成后台配置。
- 同一对用户存在未结束申请时，不允许重复申请，也不重复扣次。
- 申请成功必须在一个事务中完成：

```text
校验用户和关系状态
-> 校验认证门禁
-> 锁定额度记录
-> 扣减次数
-> 创建申请记录
-> 写入操作流水
-> 提交事务
-> 异步发送通知
```

- 通知发送失败不能回滚已成功创建的申请，但必须进入重试队列。

### 后端影响

- 申请记录、额度账户、额度流水、返还流水、幂等记录。
- 并发点击时不能扣除两次。
- 所有额度变更必须能够追溯原因。

### 确认后产物

- 申请认识规则表
- 额度流水规则
- 申请状态机：`PENDING`、`ACCEPTED`、`REJECTED`、`WITHDRAWN`、`EXPIRED`、`CANCELLED`

## 4. “双重认证”术语和认证层级

### 当前矛盾

- 当前确认只有实名认证是必须建设的认证。
- 终极版多处使用“双重认证”一词，但没有定义具体组成。

### 必须确认

需要在以下方案中选择一种：

1. 废弃“双重认证”术语，统一使用“实名认证”。
2. “双重认证” = 实名认证 + 人脸认证。
3. “双重认证” = 实名认证 + 任意一项资质认证。
4. “双重认证” = 实名认证 + 人工审核。

### 建议采用

一期废弃“双重认证”这一模糊说法，只保留：

- 未认证
- 实名认证申请中
- 实名认证通过
- 实名认证失败

二期人脸第三方接入后，再增加独立的 `FACE_VERIFIED` 状态，不建议把人脸状态隐藏在“双重认证”字段中。

### 后端影响

- 避免出现 `is_double_verified` 这种无法解释的布尔字段。
- 使用可扩展的认证项表或认证状态枚举。
- 每个功能通过权限配置声明需要哪些认证项。

### 确认后产物

- 认证术语表
- 认证项和功能门禁矩阵
- 一期/二期认证范围说明

## 5. 举报审核后的处理方式

### 当前结论

- 举报必须进入人工审核。
- 举报本身不等于立即封禁。

### 必须确认

人工审核通过后，针对不同举报类型采取什么处置：

| 举报对象 | 可能处置 |
| --- | --- |
| 帖子 | 维持、屏蔽、删除、敏感词替换 |
| 评论 | 维持、屏蔽、删除、限制评论 |
| 聊天消息 | 屏蔽消息、限制私信、封禁账号 |
| 用户 | 警告、限制单项功能、总封禁 |
| 红娘 | 下架服务、暂停接单、取消红娘资格 |
| 纸飞机 | 删除内容、冻结纸飞机、限制账号 |

还要确定：

- 举报不成立是否通知举报人。
- 举报人是否可以补充证据。
- 被举报人是否可以申诉。
- 是否允许管理员直接改判。
- 多次举报是否自动升级风险等级。

### 建议采用

举报流程独立于封禁流程：

```text
用户举报
-> 创建举报单
-> 进入人工审核
-> 审核结论
-> 执行内容处置或账号限制
-> 通知相关用户
-> 支持申诉和复审
```

建议初始处置等级：

- 轻微违规：删除内容或敏感词替换。
- 重复违规：限制发帖、评论或私信。
- 严重违规：总封禁。
- 证据不足：举报不成立，但保留记录。

### 后端影响

- 举报单、审核记录、处置记录、申诉记录、证据表。
- 审核和处置必须分开记录，不能直接把举报状态改成封禁状态。

### 确认后产物

- 举报类型字典
- 审核结论和处置动作映射表
- 申诉与复审规则

## 6. 封禁后历史内容是否展示

### 当前结论

封禁支持分级限制：

- 总封禁
- 限制发帖
- 限制评论
- 限制私信
- 限制申请认识

历史聊天保留。

### 必须确认

对不同限制类型，以下数据是否继续展示：

- 历史帖子
- 历史评论
- 历史点赞
- 历史聊天
- 个人主页
- 头像和相册
- 红娘案例

### 建议采用

| 限制类型 | 新操作 | 历史内容 |
| --- | --- | --- |
| 限制发帖 | 禁止新发帖 | 历史帖子默认保留，严重违规内容单独屏蔽 |
| 限制评论 | 禁止评论和回复 | 历史评论默认保留，违规评论可删除 |
| 限制私信 | 禁止发送新消息 | 历史聊天保留，但不允许继续发送 |
| 限制申请认识 | 禁止创建申请 | 已有匹配关系按产品规则处理 |
| 总封禁 | 禁止登录和所有用户操作 | 对外隐藏个人主页和违规内容；聊天取证保留但隐藏身份 |

总封禁不建议物理删除全部数据，避免举报证据和审计记录丢失。

### 后端影响

- 用户限制表支持多个同时生效的限制项。
- 查询内容时根据内容状态和用户状态决定是否展示。
- 封禁和内容屏蔽必须有独立状态，不能只依赖 `user.status`。

### 确认后产物

- 封禁等级和影响范围矩阵
- 历史内容可见性规则
- 解封后的数据恢复规则

## 7. 敏感词原文和审核证据保存期限

### 当前结论

- 高风险内容直接拦截。
- 一般违规内容替换为 `*`。
- 复杂或争议内容转人工审核。
- 超级管理员可以查看原文，但必须记录审计日志。

### 必须确认

需要确认以下数据分别保存多久：

- 用户提交的原始文本。
- 命中的敏感词和规则版本。
- 替换后的展示文本。
- 审核人员意见。
- 举报截图、媒体和聊天证据。
- 管理员查看原文的审计记录。
- 删除或注销后的违规记录。

### 建议采用

- 原始内容和展示内容分开存储。
- 展示内容保存当前生效版本，原始内容仅用于审核和取证。
- 每次检测记录规则版本、命中位置、处理结果和检测时间。
- 原始内容采用最小权限访问，不放入普通业务响应。
- 保存期限由合规确认；期限到期后删除原文和媒体，只保留不可还原的统计信息或必要审计摘要。

### 后端影响

- 内容表保存 `raw_content`、`display_content`、`moderation_status`、`rule_version`。
- 原文查看必须单独接口、单独权限和审计记录。
- 日志中不能输出身份证号、密码、Token、聊天原文等敏感信息。

### 确认后产物

- 内容和证据留存期限表
- 敏感词规则版本规范
- 原文查看和导出审计规范

## 8. 爆灯规则

### 已确认内容

- 有效期三天。
- 不可以撤回。
- 对方不能关闭。
- 展示在首页会员专区的个人名片中。
- 不影响推荐排序。
- 不支持退款。

### 仍需确认

- 一期是否免费开放爆灯。
- 爆灯由首页、个人主页、社区还是其他入口触发。
- 是否通知接收方。
- 同一用户对同一目标三天内能否重复爆灯。
- 爆灯失效后是否保留历史记录。
- 被拉黑或被封禁后，已生效爆灯是否立即隐藏。
- 是否需要记录展示次数和点击次数。

### 建议采用

- 一期如果没有支付和积分，建议只实现后台可配置的测试权益，不接入真实收费。
- 爆灯统一由独立服务创建，所有入口调用同一个接口。
- 同一用户对同一目标在有效期内只允许一次。
- 爆灯不改变推荐排序，只改变会员专区展示和通知状态。
- 失效后保留历史记录，但从会员专区移除。
- 拉黑或总封禁后停止对外展示，但不删除历史爆灯记录。

### 后端影响

- 爆灯表至少包含发送者、接收者、状态、生效时间、失效时间、展示位置、通知状态和创建来源。
- 不提供撤回接口，不提供退款接口。
- 使用定时任务或查询时判断自动结束，不依赖前端删除。

### 确认后产物

- 爆灯业务规则表
- 爆灯状态机
- 爆灯展示和通知接口契约

## 9. 纸飞机解锁规则

### 当前结论

- 用户侧匿名。
- 平台和超级管理员可以看到真实用户关联关系。
- 平台可以进行风控追溯。
- 会员、积分和解锁属于二期。

### 必须确认

- 解锁后可以看到哪些资料：头像、昵称、年龄、城市、完整主页、联系方式等。
- 是否需要对方同意。
- 解锁需要消耗多少积分。
- 重复解锁是否重复扣积分。
- 解锁资料是否永久有效或有有效期。
- 对方拉黑、注销或资料变更后，已解锁资料如何处理。
- 纸飞机结束后是否还能查看已解锁资料。
- 管理员查看真实身份是否需要举报单。

### 建议采用

- 一期只实现匿名身份与真实账号的内部关联，不实现积分解锁。
- 二期解锁采用一次性权益记录，同一对话同一用户只扣一次积分。
- 解锁范围默认只开放经过隐私过滤的公开资料，不开放身份证、手机号、微信号等敏感信息。
- 联系方式交换必须单独经过双方同意，不能因为资料解锁自动获得。
- 管理员和风控可追溯，但查看真实身份必须写入审计日志。

### 后端影响

- 纸飞机身份表与用户表分离，但保留内部关联。
- 解锁权益表记录解锁人、目标、会话、消耗、时间和有效期。
- 用户侧接口禁止返回真实用户 ID、手机号和内部关联字段。

### 确认后产物

- 纸飞机匿名字段规则
- 解锁资料字段白名单
- 解锁积分和有效期规则
- 平台反查审计规则

## 10. 推荐算法和隐私展示

### 必须确认

#### 推荐算法

- 一期是否使用固定规则还是直接使用 AI 算法。
- 城市、年龄、学历、MBTI、兴趣、活跃度、认证完整度的权重。
- 推荐分数是否展示给用户。
- 推荐理由是否必须能被规则解释。
- 用户修改资料后是否立即重新计算。
- 推荐列表多久刷新一次。
- 不感兴趣后多久不再推荐。

#### 隐私字段

- 距离展示精确到公里、城市还是模糊范围。
- IP 展示国家、省份、城市还是不展示。
- 收入/年薪是否展示。
- 学校和公司是否显示全称。
- 访客列表是否展示头像和主页。
- 无痕浏览是否真的不产生访客记录。

### 建议采用

- 一期使用固定规则，不依赖 AI；推荐规则和权重放配置中。
- 只返回用户允许公开且审核通过的字段。
- 距离只展示模糊范围或城市，不展示精确位置。
- IP 默认展示到省/城市级别，不返回 IP 原值。
- 收入、公司、学校等字段受隐私设置控制。
- 推荐分数一期可不直接展示，只展示可解释的推荐理由。
- 用户修改关键资料后异步刷新推荐特征，不阻塞资料保存。

### 后端影响

- 推荐查询必须统一应用认证状态、隐私设置、拉黑关系、封禁状态和资料审核状态。
- 原始 IP、精确定位和推荐特征不能直接返回前端。
- 推荐配置必须有版本，便于追踪某次推荐使用的规则。

### 确认后产物

- 推荐规则和权重表
- 用户字段展示白名单
- 隐私字段脱敏规则
- 推荐刷新和缓存策略

## 11. 一期外部服务和生产配置

### 必须确认的服务

一期涉及：

- 短信服务。
- 微信登录服务。
- 对象存储服务。
- 图片/视频/文本内容审核服务。
- 地图或 LBS 服务。

二期涉及：

- 人脸认证第三方服务。
- 支付服务。
- 会员、积分和虚拟商品服务。
- AI 红娘和 AI 推荐服务。

### 每个服务必须确认

1. 服务商和产品版本。
2. 测试环境和生产环境是否隔离。
3. API Key、Secret、证书和回调地址。
4. 请求签名和回调验签方式。
5. 超时时间和重试次数。
6. 重复回调的幂等规则。
7. 服务不可用时的降级方案。
8. 数据是否跨境、保存在哪里、保存多久。
9. 费用、调用量限制和超额处理。
10. 供应商失败时是否允许人工处理。

### 建议采用

- 所有密钥只从环境变量或配置中心读取，不能写入代码和文档示例。
- 每个第三方服务建立独立适配器，业务 Service 不直接依赖供应商 SDK 字段。
- 第三方调用记录请求流水号、供应商、响应状态和重试次数，但不得记录密钥和完整敏感内容。
- 回调接口必须验签、幂等，并允许重复通知。
- 一期 Mock 只用于开发测试，生产配置必须禁止 Mock。

### 确认后产物

- 第三方服务清单
- 环境变量清单
- 回调和重试规范
- 服务不可用降级方案
- 真实环境验收用例

## 12. 最终确认顺序

建议按以下顺序确认，避免前面的规则被后面的规则推翻：

1. 确认实名认证功能门禁矩阵。
2. 确认认证字段和资料修改规则。
3. 确认申请认识次数和状态机。
4. 废弃或定义“双重认证”术语。
5. 确认举报结论与封禁动作的对应关系。
6. 确认封禁后历史内容可见性。
7. 确认敏感内容和证据留存期限。
8. 确认爆灯一期入口、免费规则和通知规则。
9. 确认纸飞机二期解锁资料和积分规则。
10. 确认推荐权重、隐私字段和脱敏规则。
11. 确认一期第三方服务商和生产参数。

以上事项全部确认后，才能冻结一期数据库模型、接口文档、权限矩阵和验收用例。
